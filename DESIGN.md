> **Status:** Implemented — v1

## 1. Project Overview

A web-based analytics dashboard where a business user types a plain-English question, receives an auto-generated SQL query, sees the results as a table and chart, and gets an AI-written narrative insight — all in one screen.

**Target persona:** A business analyst who knows SQL but wants to move faster, or a non-technical stakeholder who cannot write SQL at all.

---

## 2. Goals & Non-Goals

### Goals
- Convert natural language questions to valid SQL using an LLM
- Execute that SQL against a local SQLite database
- Render results as a formatted table and an appropriate chart
- Generate a short insight narrative from the result data using an LLM
- Provide a handful of pre-built "example queries" so new users can explore immediately
- Keep the project fully runnable locally with no external infrastructure

### Non-Goals (for v1)
- User authentication / multi-tenancy
- Cloud deployment or hosted database
- Real-time / streaming data ingestion
- Support for multiple database backends simultaneously
- Fine-tuning or training any model

## 3. Dataset

A **synthetic e-commerce / sales dataset** stored in SQLite, generated by `data/seed.py`.

### Tables

#### `customers`
| Column | Type | Description |
|---|---|---|
| customer_id | INTEGER PK | Unique customer identifier |
| name | TEXT | Full name |
| email | TEXT | Unique email address |
| country | TEXT | Country of residence — one of 10 countries (e.g. `United States`, `India`) |
| signup_date | DATE | Account creation date (YYYY-MM-DD) |
| segment | TEXT | One of: `Enterprise`, `SMB`, `Consumer` |

#### `products`
| Column | Type | Description |
|---|---|---|
| product_id | INTEGER PK | Unique product identifier |
| name | TEXT | Product name |
| category | TEXT | One of: `Electronics`, `Apparel`, `Home`, `Books`, `Sports`, `Beauty` |
| unit_price | REAL | Listed price per unit |

#### `orders`
| Column | Type | Description |
|---|---|---|
| order_id | INTEGER PK | Unique order identifier |
| customer_id | INTEGER FK | References `customers.customer_id` |
| order_date | DATE | Date order was placed (YYYY-MM-DD) |
| status | TEXT | One of: `Completed`, `Returned`, `Pending` |

#### `order_items`
| Column | Type | Description |
|---|---|---|
| item_id | INTEGER PK | Unique line-item identifier |
| order_id | INTEGER FK | References `orders.order_id` |
| product_id | INTEGER FK | References `products.product_id` |
| quantity | INTEGER | Units purchased |
| unit_price | REAL | Price paid per unit (may differ from `products.unit_price` due to ±10% variance) |

**Actual scale (seeded):** 1,000 customers · 80 products · 5,000 orders · 15,033 order items  
**Date range:** 2023-01-01 to 2025-12-31

**Revenue formula:** `SUM(order_items.quantity * order_items.unit_price)`  
**Completed orders filter:** `orders.status = 'Completed'` (80% of orders)

## 6. Component Specifications

### 6.1 Database Layer (`src/db.py`)

```python
# Public API
def get_connection() -> sqlite3.Connection: ...
def execute_query(sql: str) -> pd.DataFrame: ...
def get_schema_description() -> str: ...
```

- Uses `sqlite3` from the standard library
- `execute_query` validates that only `SELECT` statements run (regex guard on the first keyword)
- Returns a `pandas.DataFrame` for display and charting
- `SCHEMA_DESCRIPTION` is a module-level constant — a compact plain-text description of all 4 tables, their columns, types, and useful SQL hints (revenue formula, date grouping idioms). It is injected into the LLM system prompt via `get_schema_description()`

### 6.2 AI Layer (`src/ai.py`)

```python
# Public API
def text_to_sql(question: str) -> str: ...
def generate_insight(question: str, df: pd.DataFrame) -> str: ...
```

**`text_to_sql` flow:**
1. `prompts.py` builds system prompt (schema + 10 SQL rules) + user prompt
2. Calls `openai.chat.completions.create` with model `gpt-4.1-mini`, `temperature=0.0`
3. Strips markdown fences from the response
4. Returns raw SQL string
5. Logs question, SQL, and token count to `logs/app.log`

**`generate_insight` flow:**
1. Caps DataFrame to `MAX_ROWS_FOR_INSIGHT` (default 50) rows
2. Converts to markdown table via `df.to_markdown()` (requires `tabulate`)
3. Calls OpenAI with analyst persona system prompt, `temperature=0.3`
4. Returns bullet-point insight string

**Error handling:**
- Missing API key → `EnvironmentError` with setup instructions
- Rate limit → one retry after 2 s; if still failing, raises to caller
- Any failure → caught in `app.py`, shown as a UI warning; table + chart still display

### 6.3 Prompt Templates (`src/prompts.py`)

All prompts are constants/functions here — edit to tune AI behaviour without touching business logic.

| Name | Type | Purpose |
|---|---|---|
| `SYSTEM_TEXT_TO_SQL` | constant | Schema + 10 SQL rules for the LLM (SQLite dialect, SELECT-only, LIMIT, ROUND, etc.) |
| `SYSTEM_INSIGHT` | constant | Senior analyst persona; 3–5 bullet insight rules |
| `user_sql_prompt(question)` | function | Wraps the user question for SQL generation |
| `user_insight_prompt(question, table_md)` | function | Wraps question + markdown data for insight |

### 6.4 Chart Layer (`src/charts.py`)

```python
# Public API
def pick_chart_type(df: pd.DataFrame) -> str: ...
def render_chart(df: pd.DataFrame, chart_type: str) -> go.Figure | None: ...
```

**Chart selection heuristic (5 rules, evaluated in priority order):**

| Priority | Condition | Chart type |
|---|---|---|
| 1 | Has a date/time column name + ≥1 non-ID numeric column | `line` |
| 2 | Single row + exactly 1 non-ID numeric column | `metric` (KPI card via `st.metric`) |
| 3 | Has a categorical column name + ≥1 non-ID numeric column | `bar` |
| 4 | ≥2 non-ID numeric columns | `scatter` |
| 5 | Fallback | `bar` |

**ID column filtering:** Columns whose names equal `id` or end with `_id`, `_key`, `_no`, `_num`, `_number`, `_code` are excluded from numeric value selection (e.g. `customer_id` is never used as a chart value).

**Column name hints used for classification:**

| Hint set | Keywords |
|---|---|
| Date hints | `month`, `year`, `date`, `week`, `quarter`, `period`, `day` |
| Category hints | `name`, `country`, `category`, `segment`, `status`, `product`, `customer`, `region`, `type` |

**Rendering:**
- Bar charts with > 8 rows render horizontally (sorted ascending)
- Bar charts with ≤ 8 rows render vertically (sorted descending)
- `metric` and `empty` chart types return `None` (handled separately in UI)
- Brand colour: `#6366F1` (indigo); multi-series uses Plotly Pastel palette

**Layout (single page):**

## 10. Data Flow — Step by Step

```
1.  User types question (or clicks an example pill) → clicks "Run →"

2.  app.py calls ai.text_to_sql(question)
      → prompts.py builds SYSTEM_TEXT_TO_SQL + user_sql_prompt(question)
      → openai API (gpt-4.1-mini, temperature=0) returns SQL string
      → SQL fences stripped
      → logged to logs/app.log

3.  app.py calls db.execute_query(sql)
      → regex validates SELECT-only
      → executes against analytics.db via pd.read_sql_query
      → returns pandas DataFrame
      → row count logged

4.  app.py calls charts.pick_chart_type(df)
      → applies 5-rule heuristic (ID columns excluded)
      → returns chart_type string

5.  app.py calls charts.render_chart(df, chart_type)
      → returns Plotly Figure (or None for metric/empty)

6.  app.py calls ai.generate_insight(question, df)
      → df.head(50).to_markdown() converts data to markdown table
      → openai API (gpt-4.1-mini, temperature=0.3) returns bullet insights
      → logged to logs/app.log

7.  Streamlit renders:
      - SQL via st.code(sql, language="sql")
      - KPI via st.metric  (if chart_type == "metric")
      - Table via st.dataframe(df, width="stretch")
      - Chart via st.plotly_chart(fig, width="stretch")
      - Insight via st.markdown inside styled div
      - History updated in st.session_state.history
```

## 11. Error Handling Strategy

| Scenario | Behaviour |
|---|---|
| LLM returns non-SELECT SQL | Blocked by regex; UI shows "Query blocked for safety" + raw SQL |
| LLM returns malformed SQL | `sqlite3` raises; UI shows SQL error + raw SQL for debugging |
| OpenAI API key missing | `EnvironmentError` caught in app; shows setup instructions, `st.stop()` |
| OpenAI rate limit | One retry after 2 s; if still failing, exception propagates to UI |
| Empty result set | `st.info("No data found")` — chart and insight are skipped |
| Insight call fails | `WARNING` logged; UI shows "Insights unavailable" — table + chart still render |

## 12. Logging

All runtime events go to `logs/app.log`.

```
Format:  [YYYY-MM-DD HH:MM:SS] [LEVEL] [module] message
Example: [2026-05-02 18:58:10] [INFO] [ai] text_to_sql | question='Top 5 customers by revenue'
         [2026-05-02 18:58:11] [INFO] [ai] text_to_sql | tokens_used=312 | sql=SELECT ...
         [2026-05-02 18:58:11] [INFO] [db] execute_query | sql=SELECT ...
         [2026-05-02 18:58:11] [INFO] [db] execute_query | rows_returned=5
         [2026-05-02 18:58:12] [INFO] [ai] generate_insight | question='...' | rows_sent=5
         [2026-05-02 18:58:13] [INFO] [ai] generate_insight | tokens_used=480
```

Log levels:
- `INFO` — normal flow (questions, generated SQL, row counts, token usage)
- `WARNING` — non-fatal issues (insight failure, rate-limit retry)
- `ERROR` — exceptions with full traceback

Both `FileHandler` (→ `logs/app.log`) and `StreamHandler` (→ stdout) are active.

## 15. Known Risks & Mitigations

| Risk | Status | Mitigation |
|---|---|---|
| LLM generates wrong SQL for complex joins | Active | 10-rule system prompt with schema, SQLite dialect, and revenue formula hints |
| LLM ignores SQLite dialect | Mitigated | "SQLite dialect" explicitly stated in system prompt |
| Chart heuristic picks wrong chart type | Partially mitigated | ID column filtering added; `_DATE_HINTS` and `_CAT_HINTS` sets are tunable |
| API costs during development | Low | `gpt-4.1-mini`; token usage logged per call |
| `tabulate` missing breaks insights | Fixed | Added to `requirements.txt` |
